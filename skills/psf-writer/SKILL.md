**ยินดีต้อนรับสู่ระบบช่วยเขียน Thailand-PSF**

ก่อนเริ่มร่างเอกสาร ระบบจะขอรวบรวมข้อมูลจากอาจารย์ก่อน เพื่อให้ร่างที่ได้มีความถูกต้องและตรงกับประสบการณ์จริงของอาจารย์

**ข้อมูลที่ต้องเตรียม (5 กลุ่ม)**

| กลุ่ม | รายการ | สเปค |
|------|--------|------|
| **1. ข้อมูลส่วนตัว** | ชื่อ-นามสกุล (ไทย/อังกฤษ), ตำแหน่งทางวิชาการ, คณะ, สถาบัน, สาขาวิชาหลักที่สอน, email, เบอร์โทร | ใช้สำหรับหน้าปก PSF-01 |
| **2. ประวัติการศึกษา** | วุฒิการศึกษาทุกระดับ (สาขา มหาวิทยาลัย ปีสำเร็จ) | เรียงจากสูงสุดลงมา |
| **3. ภาระงานสอน 3 ปี** | ตาราง: ปีการศึกษา / รหัส-ชื่อวิชา / ระดับ / ชั่วโมงทฤษฎี / ชั่วโมงปฏิบัติ | ย้อนหลัง 3 ปีการศึกษา |
| **4. กรณีศึกษาที่ 1** | รายวิชาที่เลือก + ข้อมูล 4 ด้าน (ดูด้านล่าง) | วิชาที่สอนจริง ย้อนหลังไม่เกิน 3 ปี |
| **5. กรณีศึกษาที่ 2** | รายวิชาที่ต่างจากกรณีที่ 1 + ข้อมูล 4 ด้าน | ต้องต่างบริบทจากกรณีที่ 1 |

**ข้อมูล 4 ด้านในแต่ละกรณีศึกษา:**
- **ด้านที่ 1 — การออกแบบรายวิชา:** ออกแบบอย่างไร ใช้หลักการใด มีนวัตกรรมอะไร + หลักฐาน/ผลที่เกิด
- **ด้านที่ 2 — การจัดการเรียนรู้:** ดำเนินการสอนอย่างไร สร้างการมีส่วนร่วมอย่างไร + หลักฐาน/ผลที่เกิด
- **ด้านที่ 3 — การวัดและประเมินผล:** วิธีประเมิน วิธีให้ feedback + หลักฐาน/ผลที่เกิด
- **ด้านที่ 4 — การพัฒนาและปรับปรุง:** เรียนรู้อะไร ปรับปรุงอะไรแล้ว/วางแผนจะทำ

> **หมายเหตุ:** "หลักฐาน" หมายถึงข้อมูลเชิงประจักษ์ เช่น คะแนนประเมินรายวิชา ผลการเรียน ข้อมูลเชิงตัวเลข หรือคำพูดจากนิสิต — ยิ่งละเอียดยิ่งดี

ระบบจะถามข้อมูลทีละกลุ่ม อาจารย์ตอบได้เลย พิมพ์สั้นหรือยาวก็ได้ ระบบจะช่วยจัดรูปแบบให้

## ขั้นตอนที่ 3: เขียน Part 3 (Reflective Writing)

### 3a. วิเคราะห์ก่อนเขียน (ทำทั้งกรณีที่ 1 และ 2)

อ่าน `teaching_cases.md` แล้ว mapping:
- dimension ร/ส/ค ที่เกี่ยวข้องกับแต่ละด้าน
- educational principles ที่ใช้สนับสนุน
- evidence ที่มีในไฟล์

แสดง mapping ให้อาจารย์ยืนยันทิศทางก่อนเขียน

### 3b. เขียน reflective writing

เขียนตาม model **What–So What–Now What** ทุกย่อหน้า:

- **What:** อธิบายสิ่งที่ทำพร้อม context ชัดเจน
- **So What:** วิเคราะห์เชิงวิพากษ์ — *ค้นพบอะไร* (ไม่ใช่แค่บอกว่าใช้เครื่องมืออะไร) มีหลักฐานสนับสนุน เชื่อมกับ educational principle
- **Now What:** เปลี่ยนแปลงการสอนอย่างไรจริง ๆ ในเทอมถัดไป — ต้องเฉพาะเจาะจง ไม่ใช่แค่ "จะพัฒนาต่อ"

**กฎที่ต้องทำทุกระดับ:**
- ทุก claim ต้องมี **(ร x.x.x)** / **(ส x.x.x)** / **(ค x.x.x)** กำกับ
- Educational principles กระจายในเนื้อหา ไม่แยก section
- ใช้เฉพาะข้อมูลใน `teaching_cases.md` ห้ามแต่งเพิ่ม
- เขียน first person ("ข้าพเจ้า...")
- **ต้องมี limitation อย่างน้อย 1 จุดต่อกรณีศึกษา**
- **กฎ: method ≠ evidence** — การอธิบายว่าใช้เครื่องมืออะไรไม่ใช่หลักฐาน หลักฐานคือสิ่งที่ค้นพบจากการใช้เครื่องมือนั้น

**เพิ่มตามระดับ:**

ระดับ 1 — เน้น 4 ด้านในห้องเรียนตัวเอง

ระดับ 2 — ต้องมีเพิ่ม:
- การแลกเปลี่ยน/ช่วยเพื่อนอาจารย์อย่างเป็นรูปธรรม (ร 1.2.x, ส 2.x.x)
- นวัตกรรมการสอนพร้อมผลที่วัดได้ (ส 2.2.x)
- การประเมินและปรับปรุงตนเองอย่างเป็นระบบ (ส 2.4.x)

ระดับ 3 — ต้องมีทั้งหมดของระดับ 2 และเพิ่ม:
- วิจัยในชั้นเรียน + ผลลัพธ์เชิงเนื้อหา (ส 3.1.x)
- นวัตกรรมการประเมิน — แยกให้ชัดจากนวัตกรรมการสอน (ส 3.3.x)
- Impact ระดับหลักสูตร — เกินกว่าห้องเรียนตัวเอง (ส 3.2.x)
- บทบาทที่ปรึกษา/พี่เลี้ยงอาจารย์ในองค์กร (ร 3.x.x, ค 3.x.x)
- ความเป็นผู้นำด้านจรรยาบรรณ (ค 3.2.x)

## ขั้นตอนที่ 4: Revision Mode (ใช้เมื่อมี PSF_review.md)

### 4a. อ่านและสรุป review

อ่าน `PSF_review.md` ทั้งหมด แล้วแจ้งอาจารย์:

> "ผลการตรวจจาก /psf-reviewer มีข้อเสนอแนะทั้งหมด [N] จุดครับ:
>
> **Part 2:**
> 1. [จุดที่ 1] — [คำอธิบายสั้น]
>
> **Part 3:**
> 1. [จุดที่ 1] — [คำอธิบายสั้น]
>
> จะ confirm ทีละจุดนะครับ"

### 4b. ถามทีละจุด

สำหรับแต่ละข้อใน review ถามอาจารย์:

> "**จุดที่ [N]:** [สรุปปัญหาที่พบ]
> reviewer แนะนำให้ปรับเป็น: [ตัวอย่างจาก PSF_review.md]
> อาจารย์ยอมรับการแก้ไขนี้มั้ยครับ? (ใช่ / ไม่ / แก้เองแล้ว)"

รอคำตอบก่อนไปจุดถัดไปทุกครั้ง

### 4c. เขียน version ใหม่

หลังได้ confirm ครบทุกจุดแล้ว:

1. ตรวจ version ล่าสุดใน folder:
```bash
ls PSF_part3_v*.md 2>/dev/null | sort -V | tail -1
ls PSF_part2_v*.md 2>/dev/null | sort -V | tail -1
```

2. เขียนเฉพาะจุดที่อาจารย์ approve:
   - Part 2 มีการแก้ → บันทึกเป็น `PSF_part2_v[N].md`
   - Part 3 มีการแก้ → บันทึกเป็น `PSF_part3_v[N].md`
   - **ห้ามแตะต้นฉบับและ version ก่อนหน้า**

3. Render version ใหม่:
```bash
pandoc PSF_part3_v[N].md -o PSF_part3_v[N].docx --reference-doc=$HOME/.hermes/psf_template.docx
```

4. แสดง diff summary:
```
📝 แก้ไขแล้ว:
✓ [section] — [สิ่งที่เปลี่ยน]
ไม่ได้แก้: [section] — อาจารย์ปฏิเสธ/แก้เองแล้ว
```

## ขั้นตอนที่ 5: Export สุดท้าย

เมื่ออาจารย์พอใจกับ version ล่าสุดแล้ว:

```bash
pandoc PSF_part2.md PSF_part3_v[N].md -o PSF_Application.docx \
  --reference-doc=$HOME/.hermes/psf_template.docx

pandoc PSF_part2.md PSF_part3_v[N].md -o PSF_Application.pdf \
  --pdf-engine=xelatex \
  --include-in-header=$HOME/.hermes/thai_pdf.tex
```

(ถ้า Part 2 มี version ใหม่ด้วย ให้ใช้ `PSF_part2_v[N].md` แทน)

แจ้งอาจารย์:
- ไฟล์ `.md` ทุกตัว — ต้นฉบับทุก version เก็บครบ
- `PSF_Application.docx` — สำหรับตรวจสอบและส่งต่อ
- `PSF_Application.pdf` — สำหรับ submit

## Revision Loop ทั่วไป

หลัง render แต่ละส่วน วน loop นี้จนอาจารย์ยืนยัน:

```
render → แจ้งอาจารย์ตรวจสอบ → รอ comment
  ├── มี comment → แก้เฉพาะจุดที่ระบุ → re-render → วนซ้ำ
  └── "ผ่าน" / "ตกลง" / "โอเค" → ไปขั้นตอนถัดไป
```

- แก้เฉพาะจุดที่อาจารย์ระบุ **ห้ามแตะส่วนอื่น**
- Re-render ทันทีทุกครั้งที่แก้
- ถ้า comment ไม่ชัดเจน → ถามให้ชัดก่อนแก้
- **ห้ามไปขั้นตอนถัดไปจนกว่าจะได้รับคำยืนยันชัดเจน**
